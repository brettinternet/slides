JAM

A site made with the JAMstack doesn't have to include all of these attributes

  • J: Any dynamic programming during the request/response cycle is handled by JavaScript, running entirely on the client. This could be any frontend framework, library, or even vanilla JavaScript.
  • A: All server-side processes or database actions are abstracted into reusable APIs, accessed over HTTPS with JavaScript. These can be custom-built or leverage third-party services.
  • M: Templated markup should be prebuilt at deploy time, usually using a site generator for content sites, or a build tool for web apps.

Why?

Why the JAMstack?

Performance

Pre-built and served over a CDN

Why wait for pages to build on the fly when you can generate them at deploy time? When it comes to minimizing the time to first byte, nothing beats pre-built files served over a CDN.

Higher Security

Reduce surface area, leverage third-party services

With server-side processes abstracted into microservice APIs, surface areas for attacks are reduced. You can also leverage the domain expertise of specialist third-party services.

Cheaper, Easier Scaling

Deploying a stack of files

When your deployment amounts to a stack of files that can be served anywhere, scaling is a matter of serving those files in more places. CDNs are perfect for this, and often include scaling in all of their plans.

Better Developer Experience

Loose coupling and separation of controls

Loose coupling and separation of controls allow for more targeted development and debugging, and the expanding selection of CMS options for site generators remove the need to maintain a separate stack for content and marketing.

Best Practices

Entire Project on a CDN

Distributed (instead of living on a single server) and pushed to the edge

Because JAMstack projects don‚Äôt rely on server-side code, they can be distributed instead of living on a single server. Serving directly from a CDN unlocks speeds and performance that can‚Äôt be beat. The more of your app you can push to the edge, the better the user experience.

Everything Lives in Git

git clone

Install deps with a standard procedure

(No databases to clone, no complex installs)

With a JAMstack project, anyone should be able to do a git clone, install any needed dependencies with a standard procedure (like npm install), and be ready to run the full project locally. No databases to clone, no complex installs. This reduces contributor friction, and also simplifies staging and testing workflows.

Take advantage of the world of modern build tools. It can be a jungle to get oriented in and it‚Äôs a fast moving space, but you‚Äôll want to be able to use tomorrow‚Äôs web standards today without waiting for tomorrow‚Äôs browsers. And that currently means Babel, PostCSS, Webpack, and friends.

Automated Builds

Markup is prebuilt, content changes won‚Äôt go live until you run another build

Because JAMstack markup is prebuilt, content changes won‚Äôt go live until you run another build. Automating this process will save you lots of frustration. You can do this yourself with webhooks, or use a publishing platform that includes the service automatically.

Atomic Deploys

Re-deploy hundreds of files (all or nothing changes)

As JAMstack projects grow really large, new changes might require re-deploying hundreds of files. Uploading these one at a time can cause inconsistent state while the process completes. You can avoid this with a system that lets you do ‚Äúatomic deploys,‚Äù where no changes go live until all changed files have been uploaded.

Instant Cache Invalidation

Ensure all changes are live

When the build-to-deploy cycle becomes a regular occurrence, you need to know that when a deploy goes live, it really goes live. Eliminate any doubt by making sure your CDN can handle instant cache purges.
